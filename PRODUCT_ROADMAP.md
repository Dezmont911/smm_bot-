# Product Roadmap / MVP -> Personal Tool -> SaaS

Документ фиксирует текущую картину продукта и спорные решения, чтобы не держать это в чате. Это не финальная спецификация. Часть пунктов актуальна для личного использования Arthur, часть может понадобиться только если продукт пойдет в SaaS.

## Текущий смысл продукта

Сейчас это не просто генератор постов. По факту это рабочий Telegram-инструмент, который совмещает:

- генератор постов;
- постер по расписанию;
- буфер и черновики;
- репост/референс-бот;
- парсер маркетплейсов;
- перекрытие РСЯ для личных каналов;
- Boost/TwiBoost для личной накрутки просмотров;
- мониторинг каналов;
- админские роли и изоляцию тестеров.

Для личного MVP это нормально: инструмент закрывает конкретные рабочие задачи владельца и пары друзей. Для SaaS это уже слишком широкая и местами токсичная смесь функций. SaaS-ядро должно быть чище.

## Главное разделение

Нужно мыслить двумя режимами продукта.

### 1. Personal / Private MVP

Это текущий режим.

Актуально:

- РСЯ-перекрытие;
- Boost/TwiBoost;
- ручные аварийные публикации;
- работа с папками каналов;
- мониторинг охватов;
- быстрые фиксы под конкретные каналы;
- изоляция тестеров;
- внутренняя диагностика;
- поддержка референсов и маркетплейсов.

Здесь можно держать узкие функции, потому что они решают реальные личные задачи.

### 2. Future SaaS

Потенциальный будущий продукт.

Вероятное ядро:

- генератор постов;
- постер;
- очередь/буфер;
- черновики;
- референсы/репосты;
- удобный календарь публикаций;
- роли/команды;
- простая аналитика;
- интеграции с Telegram;
- шаблоны постов;
- посты с продолжением через кнопку.

Скорее всего убрать или вынести в private-only:

- РСЯ-перекрытие;
- Boost/TwiBoost;
- накрутка реакций/просмотров;
- слишком специфичные личные админские функции;
- рискованные автодействия, которые обычный SaaS-пользователь не поймет.

## Ближайшие этапы для текущего MVP

### Этап 1. Стабильность продакшена

Цель: бот не должен терять посты, ломать расписание, буст, буфер и публикации.

Нужно:

- стабилизировать SSH/deploy;
- держать нормальный backup `.env`, базы, каналов и буфера;
- иметь понятный путь deploy через git/rsync/deploy_fixes;
- улучшить health-check после деплоя;
- видеть последние ошибки публикаций, генерации, Boost, WB-парсера, картинок;
- защищаться от дублей публикаций глобально;
- не публиковать новый пост слишком близко к ручному/рекламному.

### Этап 2. Контент-ядро

Цель: посты должны быть стабильно пригодными, а не просто "что-то сгенерилось".

Нужно:

- Image Router по типам каналов;
- улучшенный подбор картинок;
- антидубли картинок;
- антидубли тем;
- более жесткая валидация мусорных ответов модели;
- нормальная очистка reference-текста;
- сохранение партнерских/товарных ссылок;
- объяснение причины, почему пост отклонен.

Важное решение: для WB/marketplace картинка берется из товара/референса. Сток и генерация там не нужны.

### Этап 3. Image Router

Это отдельный важный блок.

Логика по типам:

- Marketplace/WB/Ali: только товарная картинка или медиа референса.
- Игры/CS2/Steam/Minecraft: не обычный stock как основной источник; лучше AI по строгому visual brief или игровые источники.
- Новости/факты/история/география: Wikimedia/Openverse/проверяемые публичные источники, осторожно с AI-фейками.
- Детские/локальные/образовательные: нейтральный stock или AI, без выдумывания реальных фактов/людей/офисов.
- Развлекательные каналы: stock + AI, но с антидублем.
- Reference-посты: сначала медиа донора, fallback только если медиа нет или оно выключено.

Нужно логировать:

- источник картинки;
- поисковый запрос/brief;
- почему выбран этот источник;
- почему отклонены кандидаты.

### Этап 4. Boost как private-only модуль

Для текущего MVP Boost нужен. Для будущего SaaS, скорее всего, нет.

Нужно:

- изоляция Boost-профилей;
- отдельные API-ключи для тестеров;
- просмотры и реакции;
- понятные статусы "будет работать / не будет работать";
- журнал событий по владельцу;
- не показывать чужие каналы;
- понятные причины скипа;
- защита от случайных реальных заказов.

### Этап 5. Мониторинг и алерты

Для private MVP полезно:

- мониторинг только выбранных папок/каналов;
- проверка последних постов;
- алерты по низким просмотрам;
- алерты по подписчикам раз в сутки;
- не спамить одинаковыми сообщениями;
- показывать проблемные каналы списком.

Для SaaS это можно упростить до базовой аналитики.

### Этап 6. Папки и массовые действия

Нужно довести:

- генерировать по папке;
- публиковать по папке;
- мониторить по папке;
- показывать итог операции;
- не трогать каналы вне выбранного scope;
- не ломать owner_id isolation.

### Этап 7. Роли и изоляция

Текущая логика:

- Arthur остается superadmin;
- второй админ может иметь облегченный набор функций;
- tester видит только свои каналы/свой Boost;
- чужие каналы не должны появляться в чужих меню.

Для SaaS это станет полноценной моделью tenant/team/workspace.

## SaaS-идеи на потом

### Посты с продолжением

Это важная SaaS-фича.

Суть:

- пользователь пишет или генерирует длинный пост;
- бот публикует первую часть;
- внизу кнопка, например "Читать дальше", "Узнать продолжение", "Открыть полный текст";
- текст кнопки задает пользователь;
- продолжение может открываться:
  - в боте;
  - в отдельном сообщении;
  - через deep-link;
  - через mini-app/страницу, если когда-нибудь появится.

Нужно продумать:

- где хранить продолжение;
- как считать клики;
- как не нарушить UX Telegram;
- как это работает для каналов, где бот админ;
- нужна ли аналитика по кнопкам.

### SaaS-ядро

Минимальный SaaS может быть таким:

- подключение Telegram-канала;
- генерация постов;
- очередь;
- календарь;
- референсы;
- картинки;
- посты с продолжением;
- команды/роли;
- тарифы/лимиты;
- история публикаций;
- простая аналитика.

Что не тащить в первую SaaS-версию:

- Boost;
- РСЯ-перекрытие;
- сложные private admin-функции;
- ручные серверные костыли;
- функции, которые требуют объяснять пользователю внутреннюю механику.

## Спорные вопросы

Эти пункты не считать принятыми решениями:

- нужен ли Boost в будущем SaaS;
- оставлять ли РСЯ-перекрытие только private-only или вырезать полностью;
- насколько глубоко делать мониторинг;
- какие источники картинок подключать первыми;
- стоит ли подключать Liquipedia/IGDB/Openverse;
- нужен ли отдельный web-интерфейс или пока достаточно Telegram UI;
- делать ли SaaS как новый продукт или постепенно выделять из текущего бота.

## Технические принципы

- Сначала стабилизировать ядро, потом добавлять новые фичи.
- Не чинить каждый канал отдельной заплаткой, если проблема системная.
- Все dangerous/private функции должны быть отделены от будущего SaaS-ядра.
- Любая генерация должна проходить safety/validation.
- Ссылки marketplace/reference нельзя терять.
- Картинки должны выбираться через router, а не случайно.
- Ошибки должны быть видны в UI, а не только в `journalctl`.

## Текущий приоритет

На ближайшее время разумный порядок такой:

1. Стабилизировать deploy/backup/SSH.
2. Доделать диагностику ошибок.
3. Сделать Image Router v1.
4. Довести Boost tester/reactions/statuses.
5. Улучшить reference-cleanup.
6. Довести массовые действия по папкам.
7. После этого вернуться к SaaS-ядру и постам с продолжением.

## Future: Hermes Agent как triage-слой

Идея: `nousresearch/hermes-agent` может быть внешним слоем приема жалоб, диагностики и проверки, но не вторым автономным кодером с доступом к продакшену.

Целевой сценарий:

- пользователь или тестер отправляет Hermes плохой пост, ссылку, скрин или короткое описание проблемы;
- Hermes уточняет канал, `post_url`/`message_id`, время, ожидаемое и фактическое поведение;
- Hermes делает read-only диагностику: логи, SQLite только на чтение, `check_*.py`, `git status`;
- Hermes формирует GitHub Issue или task-файл с фактами, гипотезой, логами и безопасными командами проверки;
- Codex берет задачу, чинит код, добавляет тесты, делает commit/PR;
- Hermes проверяет результат по чеклисту и возвращает короткий отчет пользователю.

Границы безопасности:

- Hermes не получает `.env`, root/VPS secrets, Telegram/OpenAI/TwiBoost ключи;
- Hermes не деплоит, не рестартит prod и не меняет БД без явного подтверждения;
- Hermes и Codex не правят одну ветку параллельно;
- первая версия Hermes-интеграции должна быть read-only + issue creation;
- любые действия, влияющие на прод, остаются за Codex/оператором до отдельного решения.

Зачем это нужно:

- меньше ручного пересыла скринов и логов в Codex;
- задачи по плохим постам, Boost, референсам и картинкам становятся структурированными;
- появляется единый формат баг-репорта для тестеров;
- можно отделить "сбор симптомов" от "правки кода".

Что не делать в первой версии:

- не давать Hermes прямой SSH/root;
- не разрешать автодеплой;
- не делать автономные правки файлов;
- не смешивать Hermes с пользовательским Telegram UI бота.
