# SAAS_SPEC — мультитенант + инвайты (закрытый тест)

Постановка для Claude Code. Цель: пустить тестеров по инвайт-коду, дать им полный
функционал на месяц, потом ограничить — при этом тестеры НЕ видят каналы/данные
владельца (Arthur), а владелец получает критические алерты себе в ЛС.

Принцип: **модульный монолит, одна БД, мягкая изоляция через `owner_id`**. Никаких
отдельных инстансов/БД на тенанта. Текущий стек не меняем.

---

## 0. Термины и роли

- **owner** — Telegram user id владельца канала (создателя). Хранится в карточке канала.
- **superadmin** — Arthur (`cfg.ADMIN_CHAT_ID`). Видит всё, получает алерты, выдаёт инвайты.
- **tester** — зарегистрированный по инвайту юзер; план `trial`/`free`/`pro`.
- Изоляция: тестер видит и трогает ТОЛЬКО свои каналы (`owner_id == его id`).

---

## 1. Данные

### 1.1 Карточка канала (channels/*.json)
- Новое поле **`owner_id`** (int). Новые каналы пишут id создателя.
- **Миграция (на VPS, не через деплой):** всем существующим карточкам проставить
  `owner_id = ADMIN_CHAT_ID`. Отдельным скриптом (как делали с расписанием), т.к.
  JSON каналов живёт на VPS и в деплое `CHANNEL_FILES=[]`.

### 1.2 Новые таблицы в БД (database.py + идемпотентная миграция в deploy_fixes)
```
users(
  user_id     INTEGER PRIMARY KEY,   -- Telegram id
  plan        TEXT DEFAULT 'trial',  -- trial | free | pro | admin
  trial_until TEXT,                  -- ISO; план trial действует до этой даты
  invited_by  TEXT,                  -- код инвайта или id пригласившего
  created_at  TEXT,
  note        TEXT
)
invite_codes(
  code        TEXT PRIMARY KEY,
  plan        TEXT DEFAULT 'trial',  -- какой план выдаёт
  days        INTEGER DEFAULT 30,    -- на сколько дней trial
  max_uses    INTEGER DEFAULT 1,
  used_count  INTEGER DEFAULT 0,
  active      INTEGER DEFAULT 1,
  created_by  INTEGER,
  created_at  TEXT
)
usage(                                -- дневные счётчики для капов бюджета
  user_id     INTEGER,
  day         TEXT,                   -- 'YYYY-MM-DD' (UTC)
  llm_calls   INTEGER DEFAULT 0,
  flux_calls  INTEGER DEFAULT 0,
  PRIMARY KEY (user_id, day)
)
```

## 2. Онбординг по инвайт-коду

- **`/start`**:
  - superadmin → как сейчас (полный доступ).
  - уже зарегистрирован → главное меню + строка статуса (план, сколько дней триала осталось).
  - не зарегистрирован → «Доступ по приглашению. Пришли инвайт-код или открой ссылку-приглашение».
- **Deep-link:** поддержать `t.me/<bot>?start=<CODE>` (Telegram отдаёт `CODE` как аргумент
  `/start`). Если код валиден и не исчерпан — регистрируем юзера: `plan=invite.plan`,
  `trial_until=now+invite.days`, `used_count += 1`, `invited_by=code`. Иначе — отказ.
- **Ввод кода текстом** (если без deep-link): принять сообщение с кодом на шаге регистрации.
- Защита: код одноразовый по умолчанию (`max_uses=1`), исчерпанный/неактивный — не пускает.

## 3. Авторизация и изоляция (КРИТИЧНО)

- Текущий `is_admin`-гейт заменить на два уровня:
  - `require_registered(user)` — для обычных команд/меню (зарегистрирован и доступ активен).
  - `is_admin(user)` — только для админских команд (генерация инвайтов, глобальная статистика).
- **Единый помощник `assert_owns(user_id, channel) -> bool`**: `channel["owner_id"] == user_id
  or is_admin(user_id)`. Вызывать в КАЖДОМ месте, где действие принимает `channel_id`
  (все ветки `handle_ui`, команды канала, генерация/постинг по запросу, референсы,
  расписание, удаление, очистка буфера). Если не владелец — «Это не твой канал», стоп.
  ⚠️ Это линчпин изоляции — пропуск хотя бы в одном колбэке = утечка в чужой канал.
- Все листинги (`load_all_channels`, экран «Мои каналы», очередь, статус) фильтруют по
  `owner_id == user_id`. Для superadmin — опционально отдельный `/admin` со всеми каналами.
- Планировщик (`poster.tick`, `run_top_up_cycle`, `import_all`) работает по ВСЕМ активным
  каналам как и раньше (он системный, не привязан к юзеру) — но при ручных действиях из
  бота проверка владельца обязательна.

## 4. Тарифы и лимиты (entitlements.py)

```
LIMITS = {
  "trial": {max_channels:5, posts_per_day:4, references:True,  ai_images:True,
            web_search:True,  bulk_add:True,  llm_calls_day:300, flux_day:30},
  "free":  {max_channels:1, posts_per_day:2, references:False, ai_images:False,
            web_search:False, bulk_add:False, llm_calls_day:40,  flux_day:0},
  "pro":   {max_channels:20,posts_per_day:8, references:True,  ai_images:True,
            web_search:True,  bulk_add:True,  llm_calls_day:2000,flux_day:300},
  "admin": {всё без лимитов},
}
```
(числа — стартовые, вынести в конфиг/таблицу, чтобы менять без правок кода.)

- `effective_plan(user)`: если `plan=='trial'` и `now > trial_until` → возвращает `free`
  (даунгрейд «на лету», без отдельного выключателя).
- `can(user, feature) -> bool` для булевых: references / ai_images / web_search / bulk_add.
- `quota(user, resource) -> int` для числовых: max_channels / posts_per_day.
- **Точки проверки (enforcement):**
  - создание канала / `/add` / `/bulk_add` → `max_channels`, для bulk → `can(bulk_add)`.
  - установка расписания (`/schedule`, UI) → ограничить число слотов ≤ `posts_per_day`.
  - тумблеры в `ui.py`: «Референсы», `image_source=ai`, `topic_source=search` → `can(...)`,
    иначе показать апселл «доступно на trial/pro».
  - генерация (`run_for_channel`, `rephrase_text`, FLUX, web-search) → перед вызовом
    проверить дневной кап `usage` владельца канала; превышен → пропустить + один алерт/день.
- **Учёт расходов:** инкремент `usage.llm_calls` на каждый вызов Claude и `usage.flux_calls`
  на каждую генерацию FLUX (по `owner_id` канала). Это и кап, и будущая аналитика расходов
  (пересекается с Tier 2 «Учёт расходов» в BACKLOG — сделать заодно).

## 5. Истечение триала (мягко, без потери данных)

- Ничего не удаляем. Ежедневный джоб (или на лету при доступе):
  - каналы сверх `free.max_channels` → `schedule_disabled=true` (пауза, не удаление);
  - платные фичи отключены через `can()` автоматически (план уже `free`).
- Сообщение юзеру: «Триал закончился. Активны N из M каналов. Продлить — …». Возврат на
  pro/trial снимает паузу.

## 7. Админские команды (только superadmin)

## 8. Честные риски (зафиксировать, не игнорировать)

- **Мягкая изоляция:** одна БД и папка `channels/`. Разделение держится на `assert_owns`
  в каждом обработчике. Для месяца доверенных тестеров — ок; не хардовая изоляция.
- **Бюджет — твой:** тестеры тратят твои Claude/FLUX. Дневные капы `llm_calls_day`/`flux_day`
  на юзера — ОБЯЗАТЕЛЬНЫ уже в триале, иначе один тестер сожжёт счёт. Это приоритет №1 фазы.
- **Telegram-лимиты:** много чужих каналов с одного бота → актуализируется лимитер из Tier 2
  (для горстки тестеров пока не критично).
- **Юзербот общий (важно):** тестеры добавляют админом только БОТА, не личный аккаунт.
  Релей референсов и `/add` по @username работают через ОДИН твой Telethon-аккаунт
  (@Artyr28Rus) как общий невидимый бэкенд. Следствия:
  - **Flood-лимиты:** все чтения/пересылки идут через один аккаунт; при параллельных
    импортах от разных тестеров поймаешь `FloodWaitError`. → В фазе 2 завести **семафор
    (`asyncio.Lock`/`Semaphore`) на ВСЕ операции юзербота** (read_candidates/forward_to_bot/
    read_channel), чтобы сериализовать обращения к Telethon-клиенту, + backoff на FloodWait.
  - **Единая точка отказа:** ограничат/забанят аккаунт — референсы и /add-по-username
    отвалятся у ВСЕХ. На масштабе нужен запасной аккаунт/пул (пока — просто помнить).
  - **Только публичные доноры:** в закрытый канал без вступления юзербот не зайдёт →
    у тестеров референсы = только публичные доноры.
  - **Перефраз = твои деньги:** сам релей дёшев, но `rephrase=on` жжёт твой Claude →
    капы из entitlements обязательны (см. раздел 4).

## 9. Фазы внедрения (можно катить по очереди)

1. **Изоляция + инвайты:** `owner_id` + миграция, таблицы `users`/`invite_codes`,

   и дневные капы LLM/FLUX + **семафор на операции юзербота** (анти-flood). (Защита денег и аккаунта.)
3. **Даунгрейд триала + админ-команды + алерты с контекстом + `/feedback`.**
