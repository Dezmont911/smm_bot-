# AGENTS.md — smm_bot (инструкции для AI-агента)

Этот файл агент читает сам при старте. **Перед любой правкой прочитай `CLAUDE.md` целиком** —
это источник правды по архитектуре, инвариантам и принятым решениям. AGENTS.md — короткая
выжимка «как мы работаем», «в какой среде» и «что нельзя ломать».

**Язык общения с пользователем (Arthur): русский.** Код: Python 3.12.
Работай как самостоятельный инженер: доводи рутинные задачи до конца (коммит + деплой + проверка),
а не останавливайся после каждого шага и не проси разрешения на очевидное.

## Среда, в которой мы работаем
- **Локалка:** Windows, `C:\Projects\smm_bot\smm_bot`. Python-лаунчер: `py script.py`.
  Терминал — PowerShell. Осторожно с кавычками: для inline-Python НЕ пиши длинные `-c "..."`
  (PowerShell их ломает) — клади скрипт в файл и запускай файл.
- **VPS:** реквизиты в локальном `.env` (`VPS_HOST` / `VPS_USER` / `VPS_KEY`) — в коде и git их НЕТ.
  Папка `/opt/smm_bot`, venv `/opt/smm_bot/venv`, сервис systemd `smm_bot.service`
  (`systemctl restart smm_bot`). БД — SQLite `/opt/smm_bot/data/content_factory.db` (WAL).
- **Деплой:** `py deploy_fixes.py` — paramiko/SFTP заливает список `FILES`, гоняет идемпотентные
  миграции БД, ставит зависимости, рестартит сервис. JSON каналов НЕ перезаписывает (там свежие
  правки из бота).
- **Логи:** `ssh -i <ключ> root@<VPS_HOST> "journalctl -u smm_bot -n 50 --no-pager"`.
- **Песочница (удалённый тест на боевом venv, без затрагивания живого бота):** скрипт в
  `scripts/_*.py` (внутри `sys.path.insert(0, "/opt/smm_bot")`), запуск:
  `set REMOTE_SCRIPT=_имя.py && py scripts\run_remote_gen.py`. Та же БД/сессия, боевого бота не трогает.

## Рабочий цикл (повторяй его, веди до конца сам)
Это НЕ «спроси разрешения на каждый шаг» — это стандартная петля: разобрался → собрал →
протестил → закоммитил → задеплоил → убедился по логам.

1. **Разберись в логике** до правок: прочитай задетые файлы, пойми поток данных и инварианты.
2. **Правки минимальные и точечные.** Без больших переписываний без явной просьбы.
3. **Компиляция:** `py -m py_compile <изменённые .py>` — синтаксис до деплоя.
4. **Тесты:** `py -m unittest tests.<модуль>` (или весь `tests/`). Для рискованных изменений —
   прогон в песочнице на боевом venv (скрипт `scripts/_*.py`, см. выше).
5. **Новый .py-модуль → ОБЯЗАТЕЛЬНО добавь его в `FILES` в `deploy_fixes.py`.** Иначе деплой
   зальёт код, который импортит несуществующий на сервере модуль → `ModuleNotFoundError` и
   краш-цикл. На этих граблях уже горели несколько раз.
6. **Коммит** осмысленной порцией, сообщение по делу, в конце строка
   `Co-Authored-By: <agent> <noreply@...>`. Перед рискованным — коммить, чтобы был откат.
7. **Деплой:** `py deploy_fixes.py`.
8. **Проверь ЧИСТЫЙ старт** по `journalctl`: есть `Бот запущен!` и `Планировщик запущен`,
   НЕТ `Traceback`. Если краш-цикл — чини сразу или откатывай, не оставляй прод сломанным.
9. **Доложи коротко:** что сделал, что проверил, результат.

Правки только в доках (`*.md`) деплоить НЕ надо — они не в `FILES` и на бота не влияют: только коммит.

## Критические инварианты — НЕ ЛОМАТЬ (сломаешь = регрессия, а не «улучшение»)
1. **Партнёрские/товарные ссылки СВЯЩЕННЫ.** `rephrase_text` отдаёт плоский текст и теряет
   `<a href>` → ссылку ОБЯЗАТЕЛЬНО переклеить через `_extract_links(text_html)`
   (`reference_importer`). Сгенерированный/переписанный marketplace-пост обязан нести РЕАЛЬНЫЙ
   `<a href>`, НИКОГДА не заглушку-слово «ссылка». Если рефакторишь генерацию/safe-слой —
   сначала проверь, что ссылка доезжает до поста (есть тест на это — гоняй его).
2. **Никакой меты в посте.** Текст модели вида «Пояснение:», «Я оставил суть…», «Если нужна
   другая версия…», «дай исходный текст» — это мусор, а не пост. Ловит `_looks_like_refusal`
   (`ai_client`) → при срабатывании берётся оригинал. Не ослабляй этот детектор.
3. **Референсы — relay по `tg_file_id`** (медиа НЕ скачиваем на диск). Поток: юзербот форвардит
   медиа в ЛС бота → `handle_userbot_forward` достаёт `file_id` и привязывает к записи
   `awaiting_media`. Не ломать forward → attach.
4. **parse_mode:** референсы = HTML (со ссылками), генерация = Markdown. `format_post_message`
   экранирует контент в HTML-карточке — иначе `<`/`&` рушат парсинг и карточка не отправляется.
5. **Изоляция тенантов:** `owner_id` + `assert_owns`/`_owns`/`scope`. Никаких кросс-тенантных
   списков и действий. Админы видят всё, тестеры — только своё.
6. **РСЯ-перекрытие персистентно** (`processed_ads.due_at`) — переживает рестарт, публикуется
   всегда при наступлении срока.
7. **Плавающие слоты:** плановый слот пропускается, если публиковали < `MIN_PUBLISH_GAP_MIN`=40 мин
   назад или ждёт перекрытие. Любая публикация пишет `last_published_utc`.
8. **Новый канал:** `schedule_disabled=true`, пустой `post_times_utc` (не постит, пока не включат
   через `/schedule`). Без дефолтных часов постинга.
9. **`callback_data` ≤ 64 байта** — не класть @handle+UUID вместе (ищем канал по post_id).
10. **Цензура:** глобальные запретки + детектор отказов + safe-гейты. Не вырезать.
11. **Добор буфера:** цель `BUFFER_TARGET`=5. Есть донор → добор с донора (+фолбэк генерация/WB);
    нет донора → генерация (marketplace → WB). `run_top_up_cycle` ПРОПУСКАЕТ донор-каналы.
    Черновики авто-добор НЕ трогает.
12. **Секреты — только в `.env`** (на VPS и локально), НЕ в git и НЕ в память.
    `*.session`, `scripts/`, `admin_settings.json` — в `.gitignore`.

## Карта файлов (кто за что; подробнее — в CLAUDE.md)
- `bot.py` — Telegram-бот: команды, `/add`, inline-колбэки, планировщик (APScheduler, UTC),
  `handle_userbot_forward` (relay медиа), мониторинг охватов, `error_handler`.
- `ai_client.py` — генерация постов, `rephrase_text`, детектор меты/отказов, лимиты полей.
- `content_generator.py` — дирижёр добора (`run_for_channel`/`run_top_up_cycle`), marketplace → WB.
- `content_safety.py` / `channel_dna.py` — safe-слой (брифы / валидация / «ДНК» канала).
  ⚠️ при правках НЕ отрубай переклейку ссылок и relay-механику.
- `reference_importer.py` — импорт референсов (relay, дедуп, фильтр рекламы, переклейка ссылок).
- `poster.py` — публикация (relay по file_id → альбом → картинка → текст), плавающие слоты.
- `buffer_manager.py` / `database.py` — очередь/черновики/перекрытия / схема SQLite.
- `ui.py` — inline-меню, карточки каналов, админ-панель.
- `deploy_fixes.py` — деплой (помни про список `FILES`!).

## Стиль
- Маленькие проверяемые правки. Сохраняй существующее поведение, если не чинишь подтверждённый баг.
- Не переписывай рабочие подсистемы целиком ради «чистоты» — здесь много неочевидных связей.
- Приоритеты ревью: 1) безопасность/изоляция, 2) рантайм-баги, 3) потеря данных,
  4) дубли публикаций, 5) надёжность планировщика, 6) контроль расходов LLM/картинок, 7) UX.
